# Migration: schema_version 1.1 → 1.2

> **Type**: MINOR (compatible; old state won't crash via `state.get(field, default)`; running the migration aligns the shoots[] fields with the schema doc)

---

## WHAT changed

### The `shoots[]` item gains 5 fields

| Field | Type | Default | Meaning |
|---|---|---|---|
| `scripts_path` | string | derived from the `prediction_file` path | the pre-shoot draft path |
| `script_consistency` | enum | `"consistent"` | `"consistent"` / `"modified"` / `"lost"`—whether the script changed when filming |
| `script_diff_pct` | int / null | `null` | proportion of changed lines 0-100; null if an old shoot never computed it |
| `v2_prediction_written` | bool | `false` | whether the prediction file already contains a `## Prediction v2` section |
| `script_hash_at_shoot` | string / null | `null` | the first-12 sha256 of `videos/<id>/script.md` |

### New concept (not in the state file, in the prediction file header)

The `Prediction Basis` field: `pre_shoot` (v1 default) / `post_shoot_pre_publish` (v2)

---

## WHY

v1.2 introduces "post-shoot script change triggers a v2 prediction re-judgment"—cheat-shoot calls `/cheat-predict — mode: v2` when it detects a script diff ≥30%, appending a `## Prediction v2` section to the original prediction file.

So cheat-retro knows "which prediction enters the calibration calculation", the shoots[] item must record whether there's a v2, and the diff strength (used to analyze the "user script-change pattern" at rubric-upgrade time).

---

## HOW (Claude steps for /cheat-migrate)

In order:

1. Read `.cheat-state.json` → parse JSON
2. Verify `schema_version == "1.1"`, otherwise error "the current state isn't 1.1, this migration isn't needed"
3. For each item in the `state["shoots"]` list:
   - if `scripts_path` is missing → derive from `prediction_file` (replace `predictions/` with `scripts/`)
   - if `script_consistency` is missing → default `"consistent"`
   - if `script_diff_pct` is missing → default `null`
   - if `v2_prediction_written` is missing → **scan the corresponding prediction file**:
     - `grep -q '^## Prediction v2' <prediction_file>` → write `true` on a hit, otherwise `false`
   - if `script_hash_at_shoot` is missing → default `null`
4. Set `schema_version = "1.2"`
5. **Atomic write**: write to `.cheat-state.json.tmp` → rename
6. Report: "✅ migration to 1.2 complete. Processed N shoots[] items."

---

## Manual fallback

Users who don't want to run `/cheat-migrate` can **leave the state entirely untouched**—all skills use `state.get(field, default)`, and the new fields auto-use defaults. But the `schema_version` field will keep showing `"1.1"`, and the SessionStart hook keeps prompting a mismatch warning.

If you want to clear the warning without running the skill, hand-edit:

```diff
 {
-  "schema_version": "1.1",
+  "schema_version": "1.2",
   ...
 }
```

The fields in shoots[] are filled on demand at the next cheat-shoot / cheat-publish trigger.
