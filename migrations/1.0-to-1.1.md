# Migration: schema_version 1.0 → 1.1

> **Type**: MINOR (compatible; old state won't crash reading defaults, but running the migration aligns the field semantics)
>
> **Status**: this is v1's first formal migration—mainly a template and example. The vast majority of early users come out of init at 1.1 already, no migration needed.

---

## WHAT changed

### Removed fields

| Field | Old value | Reason for removal |
|---|---|---|
| `mode` | "cold-start" / "calibration" binary | replaced by the `calibration_samples` integer—a hard switch is the designer's guess, a continuous integer is more accurate |
| `prediction_complexity` | "cold-start-simple" / "complete" | all predictions use the unified 7-component structure, confidence derived from calibration_samples |
| `bucket_scheme` | "ratio" / "absolute" / "absolute_with_ratio" / "percentile" | bucket boundaries auto-derived by a single algorithm (has `baseline_plays` → multiples; none → platform default) |

### Added fields

| Field | Default | Writer |
|---|---|---|
| `typical_duration_seconds` | 240 | cheat-init Q1.5 derived |
| `target_publish_cadence_days` | null (flexible cadence) | cheat-init Q1.6 derived |
| `rubric_form_mismatch` | false | cheat-init (true when content_form ≠ opinion-video) |
| `benchmark_status` | "none" | cheat-init Phase 2.5 |
| `benchmark_name` | null | cheat-learn-from |
| `benchmark_sample_count` | 0 | cheat-learn-from |
| `baseline_plays` | null | cheat-init / cheat-retro |
| `enabled_perf_adapters` | `[]` | cheat-init |
| `last_published_file` | null | cheat-publish |
| `last_retro_at` | null | cheat-retro |
| `pending_retros` | `[]` | cheat-publish / cheat-retro |
| `shoots` | `[]` | cheat-shoot / cheat-publish |

---

## WHY

v1.0's `mode` was a discrete binary, forcing a "first 5 pieces cold-start vs later calibration" choice. In reality a channel is a **continuous spectrum**—there's no qualitative change between piece 4 and piece 6. Using the integer `calibration_samples` with a derivation table (see the confidence-derivation table in [shared-references/state-management.md](../shared-references/state-management.md)) keeps the signal while removing the fake breakpoint.

Similarly `bucket_scheme` forced "how to bucket" onto the user to decide, but the user has no data to decide—handing it to an algorithm to derive from baseline_plays is more reasonable.

## HOW (Claude steps for /cheat-migrate)

In order:

1. Read `.cheat-state.json` → parse JSON
2. Verify `schema_version == "1.0"`, otherwise error "the current state isn't 1.0, this migration isn't needed"
3. Derive the new field values:
   - `typical_duration_seconds`: ask the user "how many seconds is your video's typical duration?" (default 240)
   - `target_publish_cadence_days`: ask "target publish frequency (1=daily/2=every-other-day/7=weekly/null=flexible)" (default null)
   - `rubric_form_mismatch`: check whether content_form ≠ "opinion-video" (the v1.0 field is kept) → if so, true
   - `benchmark_status`: default "none"
   - `benchmark_name`: null
   - `benchmark_sample_count`: 0
   - `baseline_plays`: scan `predictions/` for reconstructed predictions with actuals → take the median plays; if none → null
   - `enabled_perf_adapters`: `[]`
   - `last_published_file`: null (can't be inferred from 1.0 fields)
   - `last_retro_at`: null (as above)
   - `pending_retros`: scan `predictions/` for ones with `published_at` but no `## Retro` → list
   - `shoots`: scan `videos/` for ones with script.md but whose corresponding prediction has no `published_at` → construct shoot entries
4. Remove the fields: `mode`, `prediction_complexity`, `bucket_scheme`
5. Set `schema_version = "1.1"`
6. **Atomic write**: write to `.cheat-state.json.tmp` → rename
7. Report: "✅ migration to 1.1 complete. Added N fields, removed 3 fields."

## Manual fallback

Users who don't want to run `/cheat-migrate` can hand-edit `.cheat-state.json`:

```diff
 {
   "schema_version": "1.0",
+  "schema_version": "1.1",
-  "mode": "cold-start",
-  "prediction_complexity": "cold-start-simple",
-  "bucket_scheme": "ratio",
+  "typical_duration_seconds": 240,
+  "target_publish_cadence_days": null,
+  "rubric_form_mismatch": false,
+  "benchmark_status": "none",
+  "benchmark_name": null,
+  "benchmark_sample_count": 0,
+  "baseline_plays": null,
+  "enabled_perf_adapters": [],
+  "last_published_file": null,
+  "last_retro_at": null,
+  "pending_retros": [],
+  "shoots": [],
   ...
 }
```

> After hand-editing, recommend running `/cheat-status` to verify the state file can be read normally by all skills.
