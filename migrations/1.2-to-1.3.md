# Migration: schema_version 1.2 → 1.3

> **Type**: MINOR (compatible; old state won't crash via `state.get(field, default)`; running the migration aligns the fields with the schema doc)

---

## WHAT changed

### The state file gains 2 fields

| Field | Type | Default | Meaning |
|---|---|---|---|
| `last_prediction_self_scored` | bool | `false` | whether the last `/cheat-predict` went through `--skip-blind` or chose b at Phase 2.5 (trusting the main Claude's self-estimate rather than the sub-agent's blind score) |
| `last_self_scored_at` | string / null | `null` | the ISO 8601 timestamp the last `last_prediction_self_scored=true` triggered; cleared back to null on the next sub-agent run |

### New concept (not in state, in the prediction file header)

Introduces the two header fields **`BlindScored By`** + **`BlindScore Disagreement`**—see [shared-references/prediction-anatomy.md](../shared-references/prediction-anatomy.md) component 1.

Old prediction files don't have these two fields—**not force-backfilled**, cheat-retro graceful-defaults when reading.

---

## WHY

v1.3 introduces the [skills/cheat-score-blind](../skills/cheat-score-blind/SKILL.md) sub-agent as channel B—isolating the scoring action via the Task tool, letting the sub-agent see only the script + rubric_notes.md, avoiding the main conversation being contaminated by actuals/retro/user attitude.

`/cheat-predict` Phase 2 goes through the sub-agent by default; `--skip-blind` is the escape hatch. Once the escape hatch is used, a trail must be left—that's the reason `last_prediction_self_scored` exists.

`/cheat-bump` Phase 2 **mandates** the sub-agent, accepts no fallback; so a bump doesn't trigger `last_prediction_self_scored=true`.

---

## HOW (Claude steps for /cheat-migrate)

In order:

1. Read `.cheat-state.json` → parse JSON
2. Verify `schema_version == "1.2"`, otherwise error "the current state isn't 1.2, this migration isn't needed"
3. Check the `last_prediction_self_scored` field:
   - doesn't exist → default `false`
   - already exists (rare, the user hand-edited) → keep the original value
4. Check the `last_self_scored_at` field:
   - doesn't exist → default `null`
   - already exists → keep the original value
5. Set `schema_version = "1.3"`
6. **Atomic write**: write to `.cheat-state.json.tmp` → rename
7. Report: "✅ migration to 1.3 complete. Added 2 fields, no fields removed."

**No need to backfill old prediction files**—when the `BlindScored By` field is missing, treat as `unknown` (cheat-retro graceful-defaults when reading). If the user wants to mark "these old predictions are all v1.2-era inline scoring", they can manually add `BlindScored By: legacy-inline` to the file header—but it's **not done automatically in the migration**, to avoid modifying existing prediction files and triggering the immutability hook.

---

## Manual fallback

Users who don't want to run `/cheat-migrate` can **leave the state entirely untouched**—all skills use `state.get(field, default)`, and the new fields auto-use defaults. But the `schema_version` field keeps showing `"1.2"`, and the SessionStart hook keeps prompting a mismatch warning.

If you want to clear the warning without running the skill, hand-edit:

```diff
 {
-  "schema_version": "1.2",
+  "schema_version": "1.3",
+  "last_prediction_self_scored": false,
+  "last_self_scored_at": null,
   ...
 }
```

These two fields are naturally touched (written as false / null) at the next `/cheat-predict` sub-agent run, so the hand-edit is actually optional.

---

## Acceptance

After the migration:
- `jq '.schema_version' .cheat-state.json` → `"1.3"`
- `jq '.last_prediction_self_scored' .cheat-state.json` → `false` (or the kept old value)
- `jq '.last_self_scored_at' .cheat-state.json` → `null` (as above)
- run `bash hooks/session-start.sh` → no more schema-mismatch warning
